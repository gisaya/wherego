## 현재 제품 결정

- 질문은 총 7개다. 원천 3개, 검색 영향이 큰 일반 3개, 약한 재미 질문 1개를 한 세트 안에서 중복 없이 섞는다.
- 질문과 선택지의 단일 원본은 `jbg/apps/server/backend/app/resources/wherego/`다. 앱에는 로컬 질문 fallback이 없다.
- 사용자는 현재 위치 또는 직접 선택한 출발 지역을 사용한다.
- 후보 준비는 한국관광공사 KorService2와 DataLab을 사용하고 Gemini를 호출하지 않는다.
- 서버가 유효성, 거리, 하드 제약, 동일 장소 군집화, 방문자수 신호를 적용해 5~7개로 압축한다.
- Gemini 3.1 Flash-Lite는 압축 후보와 답변 메타데이터를 보고 최종 1개와 짧은 근거를 만든다. 모든 후보가 핵심 답변과 어긋날 때만 국내 장소를 직접 제안하고 관광공사 정확 일치 검색으로 재검증한다.
- 공공데이터 압축 후보가 0개여도 Gemini를 생략하지 않고 `direct` 제안을 요청한다. 직접 제안은 같은 거리·지역·반려동물 제약을 다시 통과해야 한다.
- Gemini 직접 제안이 관광공사에 없으면 `source=gemini`, `ktoVerified=false`로 명시한 자체 추천을 마지막 수단으로 반환한다. 관광공사 API 오류, 거리·지역 제약 위반, 반려동물 검증 필요 상황은 자체 추천으로 우회하지 않고 기존 후보를 사용한다.
- 실제 길찾기 API는 사용하지 않는다. 거리는 직선거리 기반 도로거리/시간 추정치이며 지도 열기는 네이버지도 검색 링크를 사용한다.
- 방문자수는 지역 단위 참고 신호다. 장소 단위 실시간 혼잡도로 표현하지 않는다.

## 현재 앱 흐름

1. `/` 또는 `/promotion` 진입 후 추천 횟수, 저장 로그인, 결제 복원과 이용권 상품을 초기 로딩 화면에서 확인
2. 초기 조회가 끝난 첫 화면에서 잔여 횟수가 0이면 광고 충전 또는 3회 이용권 구매를 즉시 실행
3. 출발 기준 선택
4. 서버 질문 7개 수신
5. 카드 선택, 질문별 배너 광고는 선택지 수와 무관하게 하단 안전영역 위에 고정
6. 마지막 답변 후 무료 후보 준비
7. 결과 전 짧은 전면광고 노출
8. 광고 `show`/`impression` 이후 Gemini 추천 시작
9. 필요 시 AI 로딩 화면
10. 결과 카드, 저장, 네이버지도, 홈 복귀. 성공 결과 2회 이상이면 한 번만 앱 리뷰 요청 CTA 표시

- `pages/index.tsx`: 일반 진입 `/`

- 결과 전 광고 안내를 `짧은 광고`로 통일했다.
- 목적지 카테고리가 있는 일반 질문은 원천 의도 다음 검색 슬롯을 보장한다. 캠핑 타입 `28`, 문화시설 타입 `14`처럼 선택 메타데이터와 일치하는 후보가 있으면 Gemini와 규칙 fallback 모두 해당 타입에서 최종 선택한다.

## 남은 위험

- 프로모션, 광고, 인앱결제, PNG 저장은 실제 최신 Toss 앱 QR에서 최종 확인해야 한다.
- 프로모션 hash 검증과 서버 지급은 최신 운영 AIT의 실제 QR에서 최초 지급·중복 차단·일시 오류 재시도를 확인해야 한다.
- 후보 0건과 추천 집중도는 3시간 QC에서 계속 본다. 표본 10건 미만의 집중도는 결함으로 단정하지 않는다.
- 이동 시간은 추정치라 실제 내비게이션 시간과 다를 수 있다.
- AI 자체 추천은 관광공사 미등록 결과이므로 실제 장소 존재 여부와 지도 검색 결과를 최신 AIT 실기기에서 확인해야 한다.
- 초기 이용정보 조회 시간과 고정 배너의 하단 안전영역·카드 겹침 여부는 최신 Toss 앱 실기기에서 확인해야 한다.

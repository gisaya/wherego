## 중복 방지

- 질문 ID 중복 금지
- 같은 tag group 중복 금지
- `mobility/accessibility`, `activity/healing_energy`, `landscape/photo`, `weather/season` 동시 선택 금지
- 원천 질문이 이미 같은 의미를 물으면 충돌하는 일반 테마 제외
- 같은 카드 안에서 의미가 거의 같은 2지선다 금지
- `cafe/bakery/cafe_street`, `photo_required/shareable`처럼 검색 결과가 같은 선택지는 유사 그룹으로 관리

중복 방지는 이전 세트와의 반복이 아니라 현재 한 세트의 변별력을 우선한다. 세트 간에는 질문 변형이 랜덤으로 다시 나올 수 있다.

## 선택지 메타데이터

목적지에 영향을 주는 선택지는 최소 다음을 가져야 한다.

```json
{
  "sourceId": "stable_id",
  "label": "사용자 문구",
  "tags": ["semantic_tag"],
  "searchHints": ["관광공사 검색어"],
  "constraints": {
    "contentPreference": "machine_readable_value"
  }
}
```

필요한 경우 아래를 추가한다.

- 관광공사 content type, cat1/cat2/cat3
- 최소·최대 거리 또는 시간
- `regionPolicy`, 허용 법정동 코드, 주소 prefix
- 반려동물, 접근성, 실내, 체류 형태 같은 하드 제약
- 후보 점수에만 쓰는 soft preference

사용자 문구와 검색 메타데이터가 반대 의미가 되면 안 된다.

## 지역 질문

- `strict`: KTO 검색과 Gemini 전 후보를 모두 허용 지역으로 제한
- `prefer`: 해당 지역을 먼저 찾되 후보 부족 시 확장 가능
- 지역 코드와 prefix는 질문 JSON이 소유한다.
- “제주/섬”처럼 명시적인 선택은 비제주 후보가 Gemini에 들어가지 않게 한다.

## 카피 기준

- 질문은 한 화면에서 읽히는 짧은 구어체
- 선택지 label은 행동이나 장소 차이가 명확해야 함
- caption은 label을 반복하지 않고 장소 예시를 짧게 보충
- 선택지의 `·` 나열은 화면에서 첫 조각만 보여도 의미가 유지되어야 함
- “형님”, 내부 기획어, AI가 사용자를 단정하는 표현 금지
- 비용, 혼잡, 반려동물 가능 여부처럼 정확도가 필요한 정보는 데이터로 검증할 수 있을 때만 단정

## 신규 문항 체크

1. 이 답변으로 관광공사 검색 또는 후보 순위가 달라지는가?
2. 반대 선택지가 실제로 다른 목적지를 만드는가?
3. 원천축이나 기존 일반 테마와 의미가 겹치지 않는가?
4. 검색 힌트가 장소명이 아니라 재사용 가능한 범주인가?
5. 하드 제약은 데이터로 검증 가능한가?
6. 2지선다 양쪽이 같은 관광지 집합으로 수렴하지 않는가?
7. 4지선다 네 개가 서로 배타적이거나 충분히 다른가?

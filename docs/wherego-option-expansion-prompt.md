# 어디고 선택지 확장 검토 프롬프트

아래 프롬프트와 `wherego-option-expansion-context.json` 파일을 다른 AI에 함께 전달한다.

```text
너는 한국 국내여행 추천 서비스의 질문 설계자이자 관광 데이터 검색 전략가다.

첨부 JSON은 토스 미니앱 "어디고"의 현재 질문 구조와 기존 원천 선택지 목록이다. 어디고는 성격 테스트가 아니라 사용자의 7개 답변으로 실제 방문할 관광지를 좁히는 앱이다. 서버는 선택지의 tags, searchHints, constraints를 한국관광공사 KorService2 검색 조건과 지역별 방문자수 가중치로 바꾸고, 최종 후보 중 1곳을 Gemini가 고른다.

목표:
1. 현재 선택지로 구분하기 어려운 여행 조건과 관광지 유형의 빈틈을 찾아라.
2. 기존 14개 일반 tagGroup에 추가할 가치가 있는 원천 선택지를 제안하라.
3. 원천질문 3축에 정말 필요한 새 질문 variant가 있으면 제안하라.
4. 기존 그룹으로 표현할 수 없는 강한 목적지 구분 축만 새 tagGroup으로 제안하라.

중요 규칙:
- 생성된 420개 질문 문장을 늘리는 것이 아니라, 질문 생성기의 재료가 되는 원천 선택지를 보강한다.
- 기존 sourceId, label, 의미 중복 그룹을 확인하고 단순한 표현 변경은 제안하지 않는다.
- 선택 하나가 관광지 유형, 검색어, 지역/거리, 접근성 필터, 날씨 대응, 혼잡도 가중치 중 적어도 하나를 바꿔야 한다.
- 2지선다는 양쪽이 명확하게 대비되어야 하며 사실상 같은 뜻이면 안 된다.
- 4지선다는 같은 축에서 서로 겹치지 않는 네 범주여야 한다.
- label은 16자 이내의 자연스러운 한국어로 쓴다.
- searchHints는 한국관광공사에서 장소를 찾는 데 쓸 수 있는 구체 명사 2~4개로 쓴다.
- 맛집이나 카페 추천 앱으로 흐르지 않게 관광지가 항상 중심이어야 한다.
- "형님", API, 데이터, 점수, 검색어 같은 내부 표현은 사용자 문구에 쓰지 않는다.
- 새 constraint를 만들면 타입과 서버 적용 방식까지 설명한다.
- 빈틈이 없는 tagGroup은 억지로 채우지 말고 건너뛴다.

검토 순서:
1. 현재 선택지의 중복과 약한 구분을 먼저 지적한다.
2. 실제 추천 후보를 더 잘 좁히는 추가 선택지만 제안한다.
3. 각 제안이 어떤 관광지 후보를 늘리거나 줄이는지 destinationImpact에 적는다.
4. JSON의 responseSchema 모양을 따라 JSON만 반환한다. Markdown과 코드펜스는 쓰지 않는다.
```

# 구조와 흐름

최종 갱신: 2026-07-10 KST

## 사용자 흐름

```text
앱 진입
-> 여행 취향 질문
-> 답변 수집
-> 여행 성향 분석
-> 여행지 후보 매칭
-> 추천 결과 카드
-> 지도 앱 연결
```

## 추천 흐름

1. 앱은 출발지 선택 뒤 서버 `POST /api/wherego/questions`에서 질문 세트를 받는다.
2. 사용자가 필수 질문 3개와 일반 질문 4개에 답한다.
3. 앱은 답변을 `tags`, `searchHints`, `constraints`로 구조화한다.
4. 서버는 선택지 메타데이터로 관광지 검색 계획을 만든다.
5. 마지막 질문 답변 직후 앱은 `POST /api/wherego/candidates`로 무료 공공데이터 후보 준비를 시작해 전환/광고 시간과 겹친다.
6. 서버는 한국관광공사 국문 관광정보 API를 분류코드와 함께 조회하고, 이동 조건/유효성 필터와 동일 장소 군집화로 의도별 6개 이하 후보를 남긴다.
7. 서버는 한국관광공사 지역별 방문자수 API로 지역 혼잡 신호를 붙여 재점수한 뒤 점수 조건에 맞는 5~7개로 압축한다.
8. 앱은 전면광고의 `show`/`impression` 이벤트가 들어온 뒤에만 `POST /api/wherego/recommend`를 호출한다. 네이티브 이벤트가 생략된 경우 `dismissed`를 한 번만 fallback으로 사용한다.
9. Gemini는 압축 후보와 선택지 메타데이터를 보고 최종 1개 장소와 이유를 고른다.
10. 서버는 최종 선택 1개에만 상세/이미지/편의 정보를 보강한다.
11. 앱은 광고 완료 후 AI 로딩 화면을 거쳐 추천 여행지 카드와 지도 연결 버튼을 보여준다.

## 질문 구조

- 필수 질문 3개: 이동 스케일, 동행/제약, 목적지 의도.
- 필수 질문은 축별로 2지선다 또는 4지선다 변형을 갖는다. 첫 화면은 이동 스케일 2지선다를 권장한다.
- 일반 질문 4개: `crowd` 1개, `mobility`/`accessibility` 중 1개, 모든 선택지가 실제 관광지 유형으로 연결되는 질문 1개, 겹치지 않는 추가 태그 그룹 1개를 선택.
- `mobility/accessibility`, `activity/healing_energy`, `landscape/photo`, `weather/season`은 한 세트에 함께 나오지 않는다. 선택된 원천질문과 주제가 겹치는 일반질문도 제외한다.
- 랜덤 질문 그룹은 `crowd`, `mobility`, `weather`, `landscape`, `activity`, `outdoor_stay`, `time_mood`, `culture_style`, `food_link`, `accessibility`, `season`, `photo`, `healing_energy`, `route_style`를 사용한다.
- `outdoor_stay`의 모든 문항에는 `캠핑·피크닉 제외`가 있다. 피크닉을 고르면 관광지 유형 `12`, 캠핑·야영·글램핑을 고르면 레포츠 유형 `28`로 검색하고, 제외를 고르면 일반 관광지 검색을 유지한다.
- 질문 세트 생성은 `jbg` 백엔드의 `POST /api/wherego/questions`가 담당한다. 앱은 서버 실패 시 앱 번들 문제은행으로 fallback한다.
- 질문은 성격 테스트가 아니라 관광지 추천 입력이다. 모든 선택지는 검색어, 거리/시간 제약, 편의조건 필터, 혼잡도 가중치 중 하나로 연결되어야 한다.

## 질문 화면

- 질문 화면은 이미지 없이 카드 선택 UI를 중심으로 구성한다.
- 2지선다는 큰 카드 2장을 세로로 배치한다.
- 4지선다는 화면 대부분을 채우는 2x2 카드 4장으로 배치한다.
- 질문 문장과 선택지는 실제 UI 텍스트로 렌더링한다.
- 출발지 선택 뒤 질문 세트 로딩 화면을 보여준다.
- 선택지 번호는 긴 문구에 눌리지 않도록 카드 우상단 고정 원형 배지로 렌더링한다.
- 하단에는 선택 중 노출되는 배너 광고 영역을 고정한다.
- 배너가 없는 첫 화면에서 전면광고를 선로딩하고, 질문 시작 시 이를 초기화하지 않는다. 선로딩 실패/타임아웃 때만 광고 게이트에서 다시 로드해 배너와 전면광고의 동시 초기 로드를 줄인다.
- 마지막 질문 선택은 무료 후보 준비만 시작하고 Gemini는 호출하지 않는다.
- 전면광고 CTA는 진행 중인 후보 준비를 재사용한다. Gemini 호출은 광고 실제 노출 이벤트 이후로 미룬다.
- 전면광고가 표시되는 동안 Gemini 분석을 함께 진행한다. 광고가 닫힌 뒤에도 분석이 남아 있으면 별도 AI 로딩 화면에서 `관광정보 후보 확인 / 방문자수 신호 비교 / AI 최종 장소 선택` 단계를 보여준다.
- AI 분석 화면과 결과 화면 맨 아래에는 각각 배너 광고를 표시하며, 전면광고 `dismissed` 이후에만 마운트한다.

## 위치와 거리 필터

- 사용자가 위치 권한을 허용하면 현재 위치를 기준으로 한다.
- 위치 권한이 없으면 사용자가 선택한 출발 지역을 기준으로 한다.
- 테스트 스크립트 기본값은 `서울시청 테스트 위치`다.
- 현재 MVP 추정식은 직선거리 x 1.35를 도로거리로 보고, 평균 45km/h 기준 왕복 시간을 계산한다.
- 왕복/편도 시간 조건에는 15분 버퍼, 거리 조건에는 10km 버퍼를 둔다.
- 거리/시간 제약은 최대값뿐 아니라 최소값도 해석한다. 예를 들어 `멀리 제대로`, `100km 넘어도 괜찮음`, `숙박도 가능` 같은 답변은 가까운 후보보다 넓은 검색 범위를 우선한다.
- `regionScope`와 `preferredRegionGroup`은 한국관광공사 지역코드 묶음으로 변환해 검색 지역을 바꾼다.
- 실제 경로 시간은 TMAP/지도 경로 API가 필요하므로 MVP에서는 추정값으로만 사용한다.

## 외부 API

한국관광공사 국문 관광정보 서비스_GW API:

- `searchKeyword2`: 키워드, 지역코드, 콘텐츠 타입으로 관광지 후보 검색
- `detailCommon2`: 주소, 좌표, 개요 등 공통정보. KorService2 현재 규격에 맞춰 `contentId`, 페이지 파라미터만 전달한다.
- `detailIntro2`: 관광지 유형 `12`와 레포츠 유형 `28`의 서로 다른 필드명을 주차, 이용시간, 휴무일, 안내 연락처로 통합한다.
- `detailImage2`: 결과 카드 이미지

한국관광공사 빅데이터 지역별 방문자수_GW API:

- `locgoRegnVisitrDDList`: 기초지자체별 일자 방문자수
- `metcoRegnVisitrDDList`: 광역지자체별 일자 방문자수
- 장소 주소에서 시/군/구를 추출해 최신 외지인+외국인 방문자수를 붙인다.
- 최신값을 같은 조회 기간의 중앙값과 비교해 `여유`, `보통`, `약간 붐빔`, `붐빔`으로 표시한다.

AI 서비스:

- Gemini Flash-Lite 사용 예정
- 관광지 검색, 후보 압축, 방문자수 조회는 서버 로직과 공공데이터 API가 먼저 처리한다.
- Gemini는 최대 7개의 점수 조건 통과 후보 중 최종 1개 선택과 설명 생성에만 사용한다.
- 여행 성향 요약
- 추천 문구 생성
- 후보지와 사용자 취향의 매칭 이유 생성

## 보안 경계

- 외부 API 키는 클라이언트에 포함하지 않는다.
- 서버 또는 API route에서 한국관광공사/Gemini를 호출한다.
- 공공데이터포털 인증키는 `.env.local`의 `PUBLIC_DATA_PORTAL_SERVICE_KEY`에서 읽는다.
- AI에는 이름, 연락처 같은 직접 식별 정보를 보내지 않는다.
- 위치 정보는 사용자가 허용하거나 직접 선택한 지역 기준으로만 처리한다.
